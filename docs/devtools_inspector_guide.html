CDP-Driven Script Development Guide

A comprehensive manual on utilizing the DevTools Inspector bridge alongside Any-Auto-Register to implement highly resilient account registration scripts that effortlessly bypass modern bot detectors.

Overview

Modern platforms like Stripe and financial institutions like Justice Federal Credit Union (JFCU) employ aggressive anti-bot protection mechanisms (e.g. Cloudflare Turnstile, hCaptcha, Akamai, and device fingerprinting). Standard automated headless browsers (e.g., vanilla Playwright, Selenium, or Puppeteer) trigger instant alerts due to exposed automation variables, specific browser profiles, and non-human event distributions.

To overcome this, our workspace leverages the DevTools Inspector Project. This system exposes a local Chrome DevTools Protocol (CDP) bridge, enabling python registration scripts to interact directly with a standard, user-profile-backed headed Chrome browser. Rather than executing native automation library commands, our automation is executed via raw HTTP requests and direct JavaScript execution.

Anti-Bot Bypassing: Because the Chrome instance is a genuine desktop browser with a real user profile, it maintains natural cookies, Canvas fingerprints, WebGL signatures, and extensions (like CapSolver), rendering typical detection systems completely ineffective.

Architecture & Flow

The system connects a lightweight python-based actuator layer directly to a Node.js bridge server, which delegates operations directly to the browser via Websockets. This structure removes any client-side automation library footprint from the host page context.

1. Actuator Layer (Python)

Registration scripts execute logic in Python. Instead of launching Playwright locally, they perform simple urllib REST calls to the bridge endpoints (e.g., /eval, /page).

2. CDP Bridge (Node.js)

An Express.js server (running at localhost:3005) that maintains a persistent WebSocket connection to Chrome over the Chrome DevTools Protocol.

Execution Pipeline

Python Script → HTTP POST to /eval → CDP Bridge → WebSocket CDP Protocol → Live Chrome Browser → Native JavaScript Execution → Response returned back to Python.

Preferred Script Design Philosophy

Our custom approach shifts away from traditional automation paradigms. Follow these primary principles to align with the user's preferred development pattern:

  1. No Client-Side Automation Drivers: Your Python registration script must not import or instantiate local browser libraries like playwright, selenium, or pyppeteer. All commands must run over the HTTP bridge.
  2. Native JavaScript Actions: When clicking buttons, toggle switches, or modal elements, do not rely on mouse movement simulation coordinates. Instead, execute element.click() directly in the browser's context via the /eval endpoint. This bypasses structural visibility blocks and reduces anti-bot telemetry checks.
  3. IIFE Scope Insulation: Raw variable declarations executed multiple times in the global context cause fatal JS engine conflicts. Wrap all JavaScript evaluated inside the page in an Immediately Invoked Function Expression (IIFE): 
    // Correct
(() => {
    const el = document.querySelector('#input-id');
    if (el) el.focus();
})();
  4. Manual Solves Over Bottlenecks: Do not attempt to automate third-party multi-factor authentication (MFA/2FA), complex bank Plaid popups, identity verification uploads, or captchas programmatically. Automatically detect their appearance, pause execution loops, and offer a fallback retry period for human solving.

React & jQuery Mobile Forms Onboarding

Standard input updates via el.value = val often fail in modern web apps. React tracks form inputs internally through state hooks, and jQuery Mobile binds custom UI widgets to raw DOM select values. To ensure your submissions are registered correctly, apply these specialized JavaScript helper methodologies:

React-Safe Inputs

To safely input text into React forms, you must call the prototype setter and dispatch bubbling input, change, and blur events so React picks up the state modification:

JavaScript (React Safe Setter) 
function fillReactInput(selector, value) {
    const el = document.querySelector(selector);
    if (!el) return 'NOT_FOUND';
    el.focus();
    try {
        const setter = Object.getOwnPropertyDescriptor(
            window.HTMLInputElement.prototype, "value"
        ).set;
        setter.call(el, value);
    } catch (e) {
        el.value = value;
    }
    el.dispatchEvent(new Event('input', { bubbles: true }));
    el.dispatchEvent(new Event('change', { bubbles: true }));
    el.dispatchEvent(new Event('blur', { bubbles: true }));
    return 'FILLED';
}

jQuery Mobile (JQM) Dropdown Selector

JQM elements require manual refresh calls after selection updates to update their visible custom widgets:

JavaScript (JQM Dropdown Setter) 
function selectJqmDropdown(selector, value) {
    const el = document.querySelector(selector);
    if (!el) return 'NOT_FOUND';
    el.value = value;
    el.dispatchEvent(new Event('change', { bubbles: true }));
    if (window.jQuery && window.jQuery.mobile) {
        try {
            window.jQuery(el).selectmenu('refresh');
        } catch (e) {}
    }
    return 'SELECTED';
}

Selector Isolation: When dealing with jQuery Mobile multipage layouts (e.g. Credit Union Loan PQ portals), always restrict your target query selectors to the active container, e.g. .ui-page-active #my-element. Otherwise, scripts will target matching selectors hidden in background views, leading to silent form failures.

Threshold Detection & Manual Interventions

Do not let your scripts crash when encountering CAPTCHAs, MFA prompts, or file uploads. Instead, programmatically check for verification keywords or specific iframes and implement a wait loop:

1. Automated hCaptcha & reCaptcha Checking

Scan the DOM structure for iframe elements loaded from CAPTCHA providers:

JavaScript 
const hasCaptcha = (() => {
    const iframes = document.querySelectorAll('iframe');
    for (const f of iframes) {
        if (f.src && (f.src.includes('hcaptcha') || f.src.includes('recaptcha'))) {
            const rect = f.getBoundingClientRect();
            const style = window.getComputedStyle(f);
            const isVisible = rect.width > 50 && rect.height > 50 && 
                              style.display !== 'none' && style.visibility !== 'hidden';
            if (isVisible) return true;
        }
    }
    return false;
})();

2. The Python Wait Loop

Inside your loop state, evaluate if a human action is required. If so, suspend automatic form submission, output a clear warning message to standard output, and loop until the condition resolves or the timeout expires:

Python Snippet 
for wait_step in range(36):  # Up to 3 minutes (36 * 5s)
    has_captcha = bridge_eval(captcha_detection_js)
    if has_captcha:
        print("  [!] CAPTCHA Detected — please solve in Chrome window!")
        time.sleep(5)
        continue
    
    # Check if page has moved on
    new_url, _ = get_page_info()
    if "/register" not in new_url:
        print("  Captcha solved successfully!")
        break
    time.sleep(5)

Complete Script Boilerplate

Use the following python script blueprint when creating a new registration script. It establishes a resilient bridge client connection, loads generated mock profiles, sets up proxies, and iterates through onboarding forms:

Python Script Blueprint (register_template.py) 
import os
import sys
import time
import json
import re
import urllib.request
import urllib.error

# 1. Bypass Proxy for Localhost Bridge calls
os.environ["NO_PROXY"] = "localhost,127.0.0.1," + os.environ.get("NO_PROXY", "")
os.environ["no_proxy"] = "localhost,127.0.0.1," + os.environ.get("no_proxy", "")

BRIDGE_URL = "http://localhost:3005"

def bridge_get(path):
    try:
        req = urllib.request.Request(f"{BRIDGE_URL}{path}")
        with urllib.request.urlopen(req, timeout=10) as resp:
            return json.loads(resp.read().decode("utf-8"))
    except Exception as e:
        print(f"  [!] Bridge GET failed: {e}")
        return None

def bridge_eval(expression):
    url = f"{BRIDGE_URL}/eval"
    data = json.dumps({"expression": expression}).encode("utf-8")
    try:
        req = urllib.request.Request(url, data=data, headers={"Content-Type": "application/json"})
        with urllib.request.urlopen(req, timeout=15) as resp:
            res = json.loads(resp.read().decode("utf-8"))
            return res.get("result")
    except Exception as e:
        print(f"  [!] Bridge JS evaluation failed: {e}")
        return None

def fill_field(selector, value):
    escaped_val = value.replace("\\", "\\\\").replace("'", "\\'").replace("\n", "\\n")
    js_expr = f"""
    (() => {{
        const el = document.querySelector('{selector}');
        if (!el) return 'NOT_FOUND';
        el.focus();
        try {{
            const setter = Object.getOwnPropertyDescriptor(window.HTMLInputElement.prototype, 'value').set;
            setter.call(el, '{escaped_val}');
        }} catch(e) {{
            el.value = '{escaped_val}';
        }}
        el.dispatchEvent(new Event('input', {{ bubbles: true }}));
        el.dispatchEvent(new Event('change', {{ bubbles: true }}));
        el.dispatchEvent(new Event('blur', {{ bubbles: true }}));
        return 'FILLED';
    }})()
    """
    return bridge_eval(js_expr) == "FILLED"

def click_element(selector):
    js_expr = f"""
    (() => {{
        const el = document.querySelector('{selector}');
        if (!el) return 'NOT_FOUND';
        el.click();
        return 'CLICKED';
    }})()
    """
    return bridge_eval(js_expr) == "CLICKED"

def main():
    # Verify Bridge Status
    status = bridge_get("/status")
    if not status or not status.get("connected"):
        print("[ERROR] CDP Bridge not connected. Start Chrome first.")
        sys.exit(1)
    
    print("[SUCCESS] Connected to CDP Chrome instance.")
    
    # Initialize form checking loop
    for step in range(1, 40):
        time.sleep(3)
        page_info = bridge_get("/page")
        url = page_info.get("url", "")
        print(f"Step {step}: {url}")
        
        # Auto fill field matches
        fill_field("#email", "john.doe@audioplexdesigns.com")
        fill_field("#password", "SuperSecure123!")
        
        # Click next triggers
        click_element("button[type='submit']")

if __name__ == "__main__":
    main()

Troubleshooting Common Roadblocks

1. "Port 9224 / 9222 already in use"

Reason: An orphaned Chrome instance, Webview2 process, or previous debug run is lingering in the background and holding the debugging port.

Solution: Kill all active browser tasks in PowerShell via Stop-Process -Name chrome -Force, or change ports in the script and devtools_inspector/.env.

2. React elements do not update after filling

Reason: Setting raw element.value bypassing the internal React state listeners.

Solution: Ensure your fill_field method calls the prototype setter and dispatches the input event as documented in our React form section.

3. JavaScript evaluates to Null / Errors out

Reason: Declarations of let or const variables with identical naming across successive /eval requests.

Solution: Always wrap browser execution snippets inside an Immediately Invoked Function Expression (IIFE) (() => { ... })().
